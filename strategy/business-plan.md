## 2. ユニットエコノミクス（Sonnet 4.5 ベース）

### 2.1 AI API コスト詳細

**モデル**: Claude Sonnet 4.5
- Input: $3/MTok (≈ ¥450/MTok)
- Output: $15/MTok (≈ ¥2,250/MTok)

**1 Chat あたりのコスト計算**:

| 項目 | トークン数 | 単価 | コスト |
|------|----------|------|--------|
| Input (system prompt + context + user message) | ~2,000 tok | ¥450/MTok | ¥0.9 |
| Output (AI response) | ~500 tok | ¥2,250/MTok | ¥1.125 |
| **合計** | **~2,500 tok** | | **≈ ¥2.0/回** |

> Note: prompt cache が有効な場合、2回目以降の system prompt 部分は 90% 削減される。実効コストは ¥1.2〜¥2.0/回。以下の計算では保守的に **¥1.5/回** を使用。

### 2.2 ティア別ユニットエコノミクス

| ティア | 月額売上 | 想定利用回数 | API コスト | 粗利 | 粗利率 |
|--------|---------|------------|-----------|------|--------|
| 🆓 Free | ¥0 | 5回/日 × 30日 = 150回 | ¥225/月 | -¥225 | — (広告で回収) |
| ⭐ Standard | ¥720 | 300回/月 (上限) | ¥450/月 | ¥270 | 37.5% |
| ⭐ Standard (実利用) | ¥720 | ~150回/月 (実態) | ¥225/月 | ¥495 | 68.8% |
| 💎 Premium | ¥1,360 | ~500回/月 (想定) | ¥750/月 | ¥610 | 44.9% |
| 💎 Premium (ライト) | ¥1,360 | ~200回/月 (ライト) | ¥300/月 | ¥1,060 | 77.9% |
| 従量 100回 | ¥360 | 100回 | ¥150 | ¥210 | 58.3% |

**重要**: Standard ユーザーの多くは 300回上限を使い切らない（実利用 150回程度と想定）。粗利率の実態は **50〜70%** と見込む。

### 2.3 Free Tier のコスト負担

```
Free ユーザー 1人あたり:
  最大利用: 5回/日 × 30日 = 150回/月
  API コスト: 150 × ¥1.5 = ¥225/月
  実利用（DAU 30%想定）: 5回 × 9日 = 45回/月
  実 API コスト: 45 × ¥1.5 = ¥67.5/月

  → 広告収入（¥50-100/MAU/月）でほぼ回収可能
```

## 4. 損益分岐点（BEP）

```
月間固定費:
  インフラ: ¥43K
  法務按分: ¥42K
  マーケティング: ¥250K
  合計: ¥335K/月

変動費/ユーザー:
  Free: API ¥67.5/月（実利用ベース）— 広告 ¥50 = net ¥17.5
  Standard: API ¥225/月（想定 150回利用）
  Premium: API ¥750/月（想定 500回利用）

Standard 粗利/人: ¥720 - ¥225 = ¥495/月
（Apple 手数料 15% 控除後: ¥612 - ¥225 = ¥387/月）

BEP 計算（Apple 手数料込み）:
  ¥335K ÷ ¥387 ≒ 866 人（Standard 課金者）
  BEP MAU = 866 ÷ 10%（課金率）≒ 8,660 MAU

→ 標準シナリオで M5〜M6 付近で BEP 達成
```

## 6. リスク分析

### 6.1 OpenClaw 依存リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|:------:|:--------:|------|
| OC サービス停止・方針変更 | 🔴 高 | 🟢 低 | API Gateway 層でのラッピングにより、LLM 呼び出し部分は差替え可能な設計。最悪の場合、直接 Anthropic API 呼び出しに切替可 |
| OC パフォーマンス劣化 | 🟡 中 | 🟡 中 | subprocess タイムアウト設定（60秒）、リトライ機構 |
| OC 料金体系変更 | 🟡 中 | 🟡 中 | 現時点で OC は LLM コスト以外のオーバーヘッドなし。変更時は直接 API に移行 |

**緩和策**: OpenClaw は「便利なラッパー」として使うが、アーキテクチャ上は差替え可能に保つ。API Gateway (FastAPI) が全てのリクエストを仲介するため、OC → 直接 API の切替はゲートウェイ層の変更のみで完了。

### 6.2 API コストリスク

| シナリオ | 条件 | 月間APIコスト | 影響 |
|---------|------|-------------|------|
| 楽観 | API 料金 50% 下落 | M12: ¥506K | 粗利率向上 |
| 標準 | 現行料金維持 | M12: ¥1,013K | 計画通り |
| 悲観 | 利用量が想定の2倍 | M12: ¥2,026K | 粗利率 10pt 低下 |
| 最悪 | 2倍 + 料金据置 | M12: ¥2,026K | ティア価格見直し検討 |

**対策**:
1. Rate Limiting で想定外の大量利用を防止
2. モデルの段階的選択（簡単な質問は軽量モデル、複雑な判断のみ Opus）
3. prompt cache の活用（同一 session での system prompt コスト 90% 削減）
4. 月次コスト監視 + 閾値アラート

### 6.3 その他のリスク

| リスク | 影響度 | 対策 |
|--------|:------:|------|
| 行政書士法抵触 | 🔴 高 | MVP 前に弁護士確認。情報提供の範囲を厳格に管理。免責表示 |

## 7. KPI 測定方式 + Post-Launch Operations

### 7.1 KPI 定義と測定手段

| KPI | 定義 | 測定手段 | 算出ロジック |
|-----|------|---------|-------------|
| DAU | 日間アクティブユーザー | Firebase Analytics `daily_active_users` | ユニーク UID / 日 |
| MAU | 月間アクティブユーザー | Firebase Analytics `monthly_active_users` | ユニーク UID / 月 |
| Chat 利用率 | AI Chat を使った DAU の割合 | カスタムイベント `chat_message_sent` | Chat DAU / DAU |
| 課金転換率 | Free → Standard/Premium | Firebase + 自社 DB | 課金者数 / 登録者数 |
| ARPU | 課金者あたり月間売上 | 自社 DB (subscriptions + purchases) | 月間課金売上 / 課金者数 |
| Churn Rate | 月間解約率 | 自社 DB (subscription_cancelled) | 解約数 / 月初課金者数 |
| NPS | ネットプロモータースコア | アプリ内アンケート（月1回サンプリング） | 推奨者% - 批判者% |
| AI 回答品質 | ユーザー満足度 | Chat 後のサムズアップ/ダウン | Good / (Good + Bad) |
| Tracker 完了率 | 自動生成された TODO の完了割合 | カスタムイベント `tracker_item_completed` | 完了数 / 生成数 |

### 7.2 Post-Launch Operations

**フィードバック収集チャネル**:
1. アプリ内: Chat 後のサムズアップ/ダウン（全 Chat に表示）
2. アプリ内: 「フィードバックを送る」ボタン（設定画面）
3. App Store / Google Play レビュー
4. SNS モニタリング（Reddit r/japanlife、Twitter/X）

**Crash Reporting**:
- Firebase Crashlytics（iOS/Android）
- エラー監視: API Gateway のエラーログ + Sentry

**迭代計画の入力ソース**:
1. KPI ダッシュボード（毎週レビュー）
2. ユーザーフィードバック集約（毎週）
3. AI 回答品質レポート（サムズダウンが多い質問パターンの特定）
4. 競合動向モニタリング（月次）
5. 知識ファイルの鮮度管理（四半期ごとにソース更新）
